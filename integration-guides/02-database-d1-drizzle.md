# 02 — Database: D1 + Drizzle ORM

Cloudflare D1 is SQLite under the hood. Drizzle has first-class D1 support (`drizzle-orm/d1`) and Better Auth ≥1.6 accepts a raw `D1Database` natively — so the two coexist without an adapter shim.

---

## Install

```bash
npm i drizzle-orm
```

That's it. No `drizzle-kit` is required at runtime — we use Better Auth's CLI to generate auth migrations and hand-write app migrations.

---

## The factory

`server/utils/db.ts`:

```ts
import { drizzle, type DrizzleD1Database } from 'drizzle-orm/d1'
import type { H3Event } from 'h3'
import * as schema from '../db/schema'

export type Db = DrizzleD1Database<typeof schema>

export const useDb = (event: H3Event): Db => {
  const env = event.context.cloudflare?.env
  if (!env?.DB) {
    throw createError({
      statusCode: 500,
      statusMessage: 'D1 binding DB not found on event.context.cloudflare.env'
    })
  }
  return drizzle(env.DB, { schema })
}
```

Use it in any handler:

```ts
export default defineEventHandler(async (event) => {
  const db = useDb(event)
  const rows = await db.query.someTable.findMany({ where: ... })
  return rows
})
```

Don't try to build the client once at module scope — `event.context.cloudflare.env` doesn't exist there.

## Schema (`server/db/schema.ts`)

Drizzle's SQLite core. The auth tables (`user`, `session`, `account`, `verification`) are owned by Better Auth — declare them here so Drizzle can resolve foreign keys and your app code can read them via Drizzle without going through Better Auth's APIs. Keep these in lock-step with `0001_better_auth.sql` (regenerate via the auth CLI; see [`03-auth-better-auth.md`](./03-auth-better-auth.md)).

```ts
import { sqliteTable, text, integer, real, index, uniqueIndex } from 'drizzle-orm/sqlite-core'

// Owned by Better Auth; mirror its CLI output.
export const user = sqliteTable('user', {
  id: text('id').primaryKey(),
  name: text('name').notNull(),
  email: text('email').notNull().unique(),
  emailVerified: integer('emailVerified', { mode: 'boolean' }).notNull(),
  image: text('image'),
  createdAt: integer('createdAt', { mode: 'timestamp_ms' }).notNull(),
  updatedAt: integer('updatedAt', { mode: 'timestamp_ms' }).notNull(),
  // Add additional columns surfaced via Better Auth `user.additionalFields`
  stripeCustomerId: text('stripeCustomerId')
})

// Your app tables…
export const item = sqliteTable('item', {
  id: text('id').primaryKey(),
  userId: text('userId').notNull().references(() => user.id, { onDelete: 'cascade' }),
  title: text('title').notNull(),
  createdAt: text('createdAt').notNull()
}, (t) => [
  index('item_userId_idx').on(t.userId)
])

export type ItemRow = typeof item.$inferSelect
export type ItemInsert = typeof item.$inferInsert
```

See `snippets/server/db/schema.ts` for the full schema with all subsystems (subscription, referral, email tables) wired in.

### Column-type cheat sheet

| Drizzle declaration                                          | SQL type    | What it stores                                 |
| ------------------------------------------------------------ | ----------- | ---------------------------------------------- |
| `text('col')`                                                | `TEXT`      | string                                         |
| `integer('col')`                                             | `INTEGER`   | number                                         |
| `integer('col', { mode: 'boolean' })`                        | `INTEGER`   | `0` / `1` ↔ `false` / `true`                   |
| `integer('col', { mode: 'timestamp_ms' })`                   | `INTEGER`   | unix-millis ↔ `Date`                           |
| `real('col')`                                                | `REAL`      | float                                          |

For ISO-string timestamps (rare), use `text('createdAt').notNull()` and pass `new Date().toISOString()` manually.

## Migrations

**Forward-only, hand-rolled SQL** under `server/db/migrations/<NNNN>_<name>.sql`. Wrangler applies them in lexical order via `wrangler d1 migrations apply <db> --local` or `--remote --env <env>`.

```
server/db/migrations/
├── 0001_better_auth.sql      ← generated by `@better-auth/cli generate`
├── 0002_app_tables.sql       ← hand-written
├── 0003_stripe.sql           ← generated, trimmed to a forward diff
├── 0004_referrals.sql        ← hand-written (optional, referral system)
├── 0005_email.sql            ← hand-written
└── 0006_<your-feature>.sql
```

### Hand-written migration

Plain SQL. SQLite-flavoured (D1 is SQLite). Forward-only — don't write `DOWN` migrations because wrangler doesn't run them.

```sql
-- server/db/migrations/0006_add_thing.sql
CREATE TABLE "thing" (
  "id" text NOT NULL PRIMARY KEY,
  "userId" text NOT NULL REFERENCES "user"("id") ON DELETE CASCADE,
  "title" text NOT NULL,
  "createdAt" integer NOT NULL
);

CREATE INDEX "thing_userId_idx" ON "thing" ("userId");
```

After writing, apply locally:

```bash
wrangler d1 migrations apply stems --local
```

If you're working in this stack, save yourself five minutes a day: just run that command proactively whenever you've touched a migration file or pulled changes — don't wait for a "no such table" error to remind you.

### Better-Auth-generated migrations

Better Auth has a CLI that diffs the schema computed from your auth config and emits the additional SQL. See [`03-auth-better-auth.md`](./03-auth-better-auth.md) — the gist is:

```bash
npx better-auth generate \
  --config server/utils/auth.cli.ts \
  --output server/db/migrations/000X_<name>.sql \
  --yes
```

The CLI emits the **entire schema**, not a diff. After running it, manually trim the file down to a forward diff (delete the `CREATE TABLE` statements for tables that already exist).

## Operations

### Local D1 lives on disk

Path: `.wrangler/state/v3/d1/miniflare-D1DatabaseObject/<hash>.sqlite`. Just a file — open it in TablePlus, Beekeeper Studio, or any SQLite GUI to poke around.

### Common queries from the CLI

```bash
# List users
wrangler d1 execute stems --local \
  --command "SELECT id, email, name, createdAt FROM user"

# Delete a user (cascades to session/account)
wrangler d1 execute stems --local \
  --command "DELETE FROM user WHERE email='test@example.com'"

# Nuke and rebuild local DB
rm -rf .wrangler/state/v3/d1
wrangler d1 migrations apply stems --local

# Production read
wrangler d1 execute stems --remote --env production \
  --command "SELECT count(*) FROM user"
```

### Export / import

```bash
# Dump prod to a file for safekeeping or to seed a new env
wrangler d1 export stems --remote --env production --output prod-backup.sql

# Restore into local
wrangler d1 execute stems --local --file prod-backup.sql
```

### Time Travel

D1 has built-in [point-in-time recovery](https://developers.cloudflare.com/d1/reference/time-travel/) up to ~30 days. It's a safety net, not a backup strategy. For anything you'd cry over, also `wrangler d1 export` on a schedule.

## Query patterns

The factory + Drizzle gives you both the query builder (`db.select()`) and the relational query builder (`db.query.table.find*`). Use the relational one for reads — it's friendlier and the type inference is better:

```ts
// Single row
const u = await db.query.user.findFirst({ where: eq(schema.user.email, email) })

// Many rows
const items = await db.query.item.findMany({
  where: eq(schema.item.userId, userId),
  orderBy: (item, { desc }) => [desc(item.createdAt)],
  limit: 50
})

// Joins / nested
const userWithSubs = await db.query.user.findFirst({
  where: eq(schema.user.id, userId),
  with: {
    subscriptions: true     // requires a relation declared in schema.ts
  }
})
```

Use the query builder for mutations and complex queries:

```ts
await db.insert(schema.item).values({ id, userId, title, createdAt: new Date().toISOString() })
await db.update(schema.item).set({ title }).where(eq(schema.item.id, id))
await db.delete(schema.item).where(eq(schema.item.id, id))

// Conditional update + count of affected rows (Drizzle's atomic CAS pattern)
const claimed = await db
  .update(schema.row).set({ rewardGrantedAt: Date.now() })
  .where(and(
    eq(schema.row.id, id),
    isNull(schema.row.rewardGrantedAt)
  ))
  .returning({ id: schema.row.id })
if (claimed.length === 0) return // someone else got there first
```

That last pattern — conditional update + `.returning()` to detect whether you won the race — is the right way to write idempotent webhook handlers. See the referral grant code in [`04-stripe-billing.md`](./04-stripe-billing.md) for a real example.

## Drizzle imports

The relational helpers (`eq`, `and`, `or`, `inArray`, `isNull`, `sql`, ...) come from `drizzle-orm` directly:

```ts
import { eq, and, or, inArray, isNull, sql, desc } from 'drizzle-orm'
```

They're **not** auto-imported by Nitro. Import per file.

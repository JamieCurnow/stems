# 00 — Architecture Overview

Before installing anything, internalise the mental model. The whole stack reuses three patterns; once those click, the rest is mechanical.

---

## Pattern 1: Per-request bindings

Cloudflare Workers don't have a global runtime env. Bindings (D1, R2, KV, Durable Objects, secrets) are passed to the worker on each request and live at `event.context.cloudflare.env`. There is **no top-level `process.env`** to import from.

So every server-side utility is a *factory function* that takes the H3 event:

```ts
//                v request-scoped event
export const useDb = (event: H3Event) => {
  const env = event.context.cloudflare?.env
  if (!env?.DB) throw createError({ statusCode: 500, statusMessage: 'D1 binding missing' })
  return drizzle(env.DB, { schema })
}
```

Same shape for `useStripe(event)`, `useResend(event)`, `serverAuth(event)`. If you find yourself wanting `import { db } from '~~/server/utils/db'`, stop — you can't, and trying to work around it (top-level `let db: Db | null = null`) creates cross-request leaks.

This is why Better Auth is also built per-request:

```ts
export function serverAuth(event: H3Event) {
  const env = event.context.cloudflare?.env
  return betterAuth({
    database: env.DB,
    secret: env.BETTER_AUTH_SECRET,
    // ...
  })
}
```

The cost is one Better Auth construction per request. That's a few hundred microseconds — negligible compared to a D1 round-trip.

## Pattern 2: `wrangler.jsonc` is the single source of truth

Nitro can technically write its own `wrangler.json` from inline `nitro.cloudflare.wrangler` config in `nuxt.config.ts`. **Don't use it.** That path made the multi-env setup unmanageable, and Nitro's `cloudflareDev` plugin reads from a real `wrangler.jsonc` at the project root anyway.

So:

```ts
// nuxt.config.ts
nitro: {
  preset: 'cloudflare-module',
  cloudflare: {
    deployConfig: false, // <- this is the load-bearing line
    nodeCompat: true
  }
}
```

And then `wrangler.jsonc` carries everything:

```jsonc
{
  "name": "stems",
  "main": ".cloudflare/worker.ts",
  "compatibility_date": "2025-07-15",
  "compatibility_flags": ["nodejs_compat"],

  // Top-level = local dev (Miniflare reads this).
  // env.staging + env.production = deployed envs.
  "d1_databases": [{ "binding": "DB", "database_name": "stems", ... }],
  "r2_buckets": [...],
  "vars": {...},

  "env": {
    "staging":    { "name": "stems-staging", ... },
    "production": { "name": "stems",         ... }
  }
}
```

`wrangler deploy --env <name>` does **not** merge top-level into the env block — each env block is self-contained. So if a binding should be the same across all envs, declare it once at the top and let the env-init tooling (or you, manually) propagate it into each env block.

## Pattern 3: The DO wrapper

Nitro's `cloudflare-module` preset emits a worker that only exports `default { fetch }`. But Durable Object classes have to be **named exports** so wrangler can bind them.

The fix is a one-line wrapper at `.cloudflare/worker.ts`:

```ts
// @ts-ignore - generated by `nuxt build`
import nitroHandler from '../.output/server/index.mjs'

export { EmailScheduler } from '../server/durable-objects/EmailScheduler'
export default nitroHandler
```

`wrangler.main` points at this wrapper, not at `.output/server/index.mjs` directly. Wrangler's esbuild bundles both in.

Local dev (`nuxt dev`) runs Miniflare against the Nitro bundle directly, **without** the wrapper — so DO bindings are unavailable in dev. The `scheduleEmail` helper detects this and falls back to a D1-only queue path, which is drained by a cron-runnable endpoint. This dev/prod split is intentional: it keeps dev fast (no DO startup cost) and gives you a clean diagnostic when something doesn't work in prod.

## Request flow

A typical authenticated POST through the system:

```
Browser
  │  cookie: __Secure-better-auth.session_token=...
  ▼
Cloudflare Worker (Nitro module)
  │  event.context.cloudflare.env = { DB, RESEND_API_KEY, ... }
  ▼
server/api/foo.post.ts
  │
  ├── const user = await requireUser(event)
  │      └── serverAuth(event).api.getSession({ headers })
  │             └── Better Auth reads/writes D1 via env.DB
  │
  ├── const db = useDb(event)
  │      └── drizzle(env.DB, { schema })
  │
  └── const stripe = useStripe(event)
         └── new Stripe(env.STRIPE_SECRET_KEY, { httpClient: createFetchHttpClient() })
```

Webhook flow (Stripe webhook to `/api/stripe/webhook`):

```
Stripe
  │  Stripe-Signature: t=...,v1=...
  ▼
Cloudflare Worker
  │
  ▼
server/api/stripe/webhook.post.ts
  │
  ├── const sig = getHeader(event, 'stripe-signature')
  ├── const raw = await readRawBody(event, false)
  │
  └── const event = await stripe.webhooks.constructEventAsync(
        raw, sig, env.STRIPE_WEBHOOK_SECRET, undefined, stripeCryptoProvider
      )                                                  ^^^^^^^^^^^^^^^^^^^^
                                                         Stripe.createSubtleCryptoProvider()
                                                         — Workers don't have Node crypto
```

Email scheduling flow (e.g. trial reminder):

```
Trigger (auth hook / Stripe webhook / form)
  │
  ▼
scheduleEmail(event, { emailId, sendAt, dedupeKey, ... })
  │
  ├── INSERT INTO scheduledEmail (audit row)
  │
  └── EMAIL_SCHEDULER.idFromName(dedupeKey).fetch('/schedule', { body })
         │
         ▼
       Durable Object instance
         │  state.storage.put('payload', ...)
         │  state.storage.setAlarm(sendAt)
         ▼
       <time passes>
         │
         ▼
       alarm()
         │
         ├── audience check (D1 — suppression + per-category prefs)
         ├── renderEmail(emailId, props, ctx)
         └── resend.emails.send(...)
```

Each scheduled email is its own DO instance, named by `dedupeKey`. Calling `schedule` twice with the same key targets the same DO → no-op. Each DO has exactly one alarm. Cloudflare retries the alarm on throw with built-in backoff; the DO caps at 5 attempts internally to defang poison messages.

## File layout

```
project-root/
├── nuxt.config.ts                  ← Nitro preset, compat date, modules, PWA, fonts
├── wrangler.jsonc                  ← Cloudflare bindings (dev + staging + prod)
├── .cloudflare/
│   └── worker.ts                   ← DO wrapper (re-exports DO classes + Nitro handler)
├── app/
│   ├── app.vue                     ← Top-level chrome
│   ├── pages/                      ← File-routed pages
│   ├── components/                 ← Auto-imported by Nuxt
│   ├── composables/                ← useAuth, useSubscription, …
│   ├── middleware/                 ← auth.ts, subscription.ts, admin.ts
│   └── utils/
│       └── auth-client.ts          ← Better Auth Vue client
├── server/
│   ├── api/                        ← HTTP handlers (Nitro auto-routes)
│   │   ├── auth/[...all].ts
│   │   ├── billing/, stripe/, admin/, cron/, email/, resend/, ...
│   ├── routes/                     ← Public (login-less) routes — /r/[code], /email/unsubscribe
│   ├── db/
│   │   ├── schema.ts               ← Drizzle schema
│   │   └── migrations/             ← Forward-only SQL files
│   ├── durable-objects/
│   │   └── EmailScheduler.ts
│   ├── emails/                     ← Template files + registry
│   ├── types/
│   │   └── cloudflare.d.ts         ← Bindings shape, augments H3EventContext
│   └── utils/                      ← useDb, useStripe, useResend, requireUser, ...
├── shared/                         ← Crosses SSR boundary (types + constants)
│   └── utils/                      ← e.g. constants.ts
└── public/                         ← Static assets + PWA icons
```

That's the whole map. Everything else in this guide is variations on these three patterns.
